# ConditionIQ Security Remediation — Master Summary

**Date range:** 2026-07-06 (audit) through 2026-07-07 (full remediation sequence)
**Status:** All numbered steps (Emergency fix through Step 12) complete and confirmed run against the live database. Testing and a small set of deferred follow-up items remain outstanding — see the final section.

This document is a synthesized closure summary of the full engagement. The authoritative, step-by-step technical record — exact SQL, exact diffs, exact reasoning for every decision — remains `SECURITY_REMEDIATION_LOG.md`. This file compresses that record into one reference document; if the two ever disagree on a technical detail, the log is correct.

---

## Overall context

This remediation sequence followed a full read-only security audit of the ConditionIQ codebase and Supabase backend conducted on 2026-07-06, which produced 17 numbered findings ranging from Critical to Low severity, plus one blocking item (three RLS helper functions — `is_admin()`, `is_platform_owner()`, `get_my_company_id()` — that gated a large fraction of the schema's access control but had never been read). Fixing began with that blocking item: reading those functions' actual bodies surfaced a far more severe, previously-unknown issue not on the original list at all — a self-service privilege escalation where any authenticated user could grant themselves full platform-owner access with a single ordinary database call, bypassing every other access control in the application. That was fixed immediately, out of numbered order, as an emergency step, before the planned sequence resumed. From there, the 17 original findings were worked through as a mix of dedicated numbered steps (Steps 2 through 12) and a couple of split/merged steps (3a/3b for the two halves of the worst original finding, 2.5 and 3.5 inserted as they were discovered to be prerequisites or closely related follow-on work). Every step followed the same discipline: re-verify the live database state rather than trust the original audit snapshot, trace actual application code to determine what a fix would break before proposing it, and — where a fix required both a database migration and a code change — determine and state the correct deploy ordering explicitly rather than assume either direction was safe. That discipline caught several places where the obvious fix would have silently broken a real, working feature (most notably the guest/anonymous inspection flows), and also surfaced a number of pre-existing, unrelated application bugs that were fixed only because later steps needed the affected flows to actually work in order to validate a security fix against them.

| #3 | Broad `auth.role() = 'authenticated'` policies on 9 further tables (`inspection_audit_log`, `inspection_snapshots`, `inspection_custody`, `inspection_queue`, and 5 `crm_*` tables) | Critical | Any logged-in user, any company, could read/write these tables — either no scoping at all, or a broad policy silently overriding an already-correct one sitting next to it. | Each table now has real scoping: join-through-`vehicle_inspections` company checks for the three `inspection_*` tables, the pre-existing correct `queue_isolation` left as the sole policy on `inspection_queue`, and admin-only policies on all five `crm_*` tables (confirmed to have no per-user concept in the schema). `inspection_custody`'s active guest-write path was additionally routed through the new authorization module (mirroring finding #2's fix) rather than left to a policy alone. | #8 / #10 | `inspection-photos` table + storage bucket — public bucket, anonymous unrestricted upload, no ownership check on photo records | Critical | Bucket was `public: true`, no size/type limit, with policies allowing anyone to list/read every company's photos and anonymously upload arbitrary files. ## What changed for users/customers

**Nothing became harder to use.** Every fix in this sequence was designed and verified (via code tracing, not assumption) to preserve every legitimate workflow — normal staff usage, the guest/anonymous field-inspector flows (dispatch links, FMC dispatch, "complete later" links, resume-after-disconnect, cancel), and the admin panel all continue to work exactly as they did before, as far as could be confirmed by reading the code paths involved. No UI changed. No feature was removed except the AI email-drafting buttons in the internal sales CRM (finding #1) — confirmed not in active use before removal, and it never touched customer-facing product surfaces at all.

**What customers can no longer do, because they were never supposed to be able to:**
- Grant themselves owner/admin access to the platform.
- Read, modify, or delete another company's inspections, lot layout, background images, invoices, inspection reports, feature-flag entitlements, audit trail, or vehicle photos.
- Enable paid features (Lot Map, Lot Billing, White Label, Dispatch) without actually paying for them.
- Enumerate every company's public share-link tokens or invoice/report PDFs by guessing a UUID or sequential invoice number.
- Spam the public contact form without any practical limit.

**A small number of pre-existing, unrelated bugs were fixed as a side effect**, not because they were security issues, but because later steps needed the affected flows working in order to validate a fix:
- The "staff pre-creates an inspection, guest completes it later" flow (`/complete/[token]`) was silently broken — it read a database column that never existed. Now fixed.
- The FMC (fleet-management-company) dispatch flow was similarly broken — it read a different nonexistent column, meaning it likely never successfully created an inspection at all. Now fixed.
- Two other pre-existing, low-impact bugs were found and **left unfixed**, logged as backlog items: `storage_vehicles` inventory records likely never get created for guest-completed inspections (silent RLS failure, uninvestigated before this audit), and the admin panel's "Company Notes" feature has likely never worked at all (the underlying table has row-level security enabled with zero policies, blocking everyone — including admins).

## What's still outstanding or deferred

**1. Step 3b's six-scenario staging test suite — written, never run.** This was the guest-access authorization redesign for the worst finding in the audit (#2). A full manual test plan was written (fresh start-to-completion, resume after a dropped connection, "start fresh" over a stale in-progress inspection, mid-inspection cancel, the `/complete/[token]` flow, and confirming an expired/invalid token is correctly rejected) but per your explicit instruction, all testing across this entire sequence was deferred to one consolidated pass at the end rather than gated per-step. **This is the single most important remaining action item** — it's the highest-stakes code change in the whole sequence (a new authorization module touching 7 files) and has not been exercised end-to-end.

**2. Next.js major-version upgrade (finding #13) — deliberately out of scope for this entire sequence, a separate project.** Currently on 14.2.35; latest stable is 16.2.10. The advisory count underneath this one entry has grown from 5 to 14 distinct CVEs since the original audit (including newer ones: XSS in App Router apps using CSP nonces, SSRF via WebSocket upgrades, cache-poisoning issues, a middleware/i18n bypass). No action taken; noted as context for whoever schedules this.

**3. The accumulated `*-actions.ts` ownership-check sweep.** The pattern fixed explicitly in `lib/admin-actions.ts` (#5) and `lib/lot-server-actions.ts`'s `lot_spots` functions (#6) — service-role/admin-client calls with no ownership or authorization check of their own — was flagged by the original audit as likely recurring elsewhere, and confirmed to recur every time it was checked. Never swept as its own dedicated pass. Known/confirmed instances still open:
   - `dispatch-actions.ts` (never traced)
   - `vehicle-events-actions.ts` (never traced)
   - `bulk-invoice-actions.ts` (never traced)
   - Remaining, untouched parts of `inspection-server-actions.ts` beyond what Steps 3b/4/9 already fixed
   - Five sibling functions in `lib/lot-server-actions.ts`: `createLotShapeAction`, `updateLotShapeAction`, `deleteLotShapeAction`, `removeLotBackgroundAction`, `saveLotBillingDefaultsAction` — same admin-client-no-check shape as the three functions fixed in Step 8, confirmed present, not fixed
   - `lib/contact-actions.ts`'s `getContactRequests` and `updateContactRequestStatus` — not touched when `submitContactRequest` was fixed in Step 11
   - `components/billing/billing-dashboard.tsx`'s plan-upgrade request — inserts directly into `contact_requests` from the browser with no rate limit or auth check; needs restructuring into a server action before any check has anywhere to live, not just an added check

**4. Small, already-identified, not-yet-scheduled fixes** (none are security vulnerabilities on their own — each is either a data-integrity bug or an architectural inconsistency):
   - `company_notes` has RLS enabled with zero policies, meaning the admin panel's "Company Notes" feature is almost certainly non-functional for everyone, including legitimate admins. Needs one `is_platform_owner()`-gated policy added.
   - `storage_vehicles` inventory records likely silently fail to be created/updated for guest-completed inspections — the writing function never checks the RLS-rejection error it gets back. Needs either error-surfacing or routing through an authorized admin-client path, consistent with how similar guest-write issues were fixed elsewhere in this sequence.
   - Two confirmed instances of code writing to a database column that doesn't exist, both silently failing: `buildCustodyRecord` writes `completed_at` where the real column is `signed_at`; `completeFMCInspection` writes `completed_at` to `fmc_inspection_requests`, which has no such column (its `status` field is set correctly in the same call, so this is lower-impact, but still a dead write).
   - Architectural inconsistency: `is_platform_owner()`/`is_admin()` check `user_profiles.role = 'owner'`, while `/admin/*` page access is gated by a different column, `platform_role = 'super_admin'`. Today these are always the same one account, so there's no live issue — but if the owner ever grants panel access to a second person without also making them the literal "owner," that person could load admin pages and then have every query silently rejected. Worth deciding on one consistent axis before that scenario comes up.
   - The `auth.users` signup-trigger blind spot: no code path was found that legitimately updates `user_profiles.company_id` after row creation, which is reassuring for the emergency-fix trigger's coverage, but this was confirmed by searching application code only — a signup-time database trigger outside the tracked migration files was never directly ruled out via its own inspection. A follow-up `pg_trigger` query against `auth.users` would close this definitively.

**Everything else** — every one of the original 17 audit findings that had its own dedicated remediation step, plus the emergency privilege-escalation fix that predated the numbered sequence — is fixed, deployed, and independently confirmed against the live database as of this summary.
